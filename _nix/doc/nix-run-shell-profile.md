# 请对比 nix shell, nix run 和 nix profile install 的使用场景和对系统的持久性影响。

简单来说，这三个命令代表了 Nix 处理软件包的**三种生存周期**：**瞬时（运行完即消失）**、**临时（会话结束即消失）**、以及**持久（手动卸载才消失）**。

下表直观地对比了它们的核心区别：

| 特性               | `nix run`                      | `nix shell`                         | `nix profile install`           |
| ------------------ | ------------------------------ | ----------------------------------- | ------------------------------- |
| **持久性**         | **瞬时**。仅在命令执行期间存在 | **临时**。仅在当前 Shell 会话中有效 | **持久**。直到你手动卸载        |
| **对 PATH 的影响** | 无。直接执行二进制文件         | 局部。修改当前 Shell 的环境变量     | 全局。修改用户环境的 `PATH`     |
| **主要用途**       | 运行一次性工具（如初始化脚本） | 开发调试、临时使用多个工具          | 安装常用工具（如 `git`, `vim`） |
| **资源占用**       | 运行完后环境即释放             | 退出 Shell 后环境即释放             | 始终占用用户环境位置            |

---

## 1. `nix run`：极简主义的“快闪”

当你只需要执行一个命令，且不想让这个命令干扰你的 Shell 环境时使用。

- **场景**：你想生成一个镜像、初始化一个项目模板，或者偶尔检查一下 `hello` 命令。
- **副作用**：几乎为零。它会下载包到 `/nix/store`，但不会在任何地方留下配置或环境变量。
- **例子**：`nix run nixpkgs#cowsay -- "Hello"`

## 2. `nix shell`：临时的“工具箱”

当你需要在一个特定的环境下工作一段时间，且需要多个工具配合时使用。

- **场景**：你正在调试一个 Go 程序，需要临时用到 `mockgen`、`go` 和 `golangci-lint`。
- **副作用**：只影响当前的终端窗口。一旦你输入 `exit` 或关闭窗口，这些工具就会从你的 `PATH` 中消失。
- **注意**：它**不会**像 `nix-shell` 那样默认去寻找 `shell.nix`（除非你显式指定），它更倾向于快速合成环境。

## 3. `nix profile install`：传统的“安装”

这是最接近 `apt install` 或 `brew install` 的命令。

- **场景**：你需要每天都用到的基础工具，比如 `fzf`、`ripgrep`。
- **副作用**：它会在你的用户目录（通常是 `~/.nix-profile`）下创建符号链接。即便重启电脑或新开终端，这些命令依然可用。
- **管理**：需要通过 `nix profile list` 查看，`nix profile remove` 卸载。

---

> ### 💡 专家视角：关于“垃圾回收”的真相
>
> 无论你用哪个命令，Nix 都会把文件下载到 `/nix/store`。虽然 `nix run` 的环境是“瞬时”的，但**二进制文件本身并不会在运行完后立刻删除**。
> 如果你想彻底清理这些不再使用的包，需要定期运行：
> `nix-collect-garbage -d`
> 只有这样，那些没有被 `nix profile`（或其他根节点）引用的包才会被物理删除。
