# .env with .envrc

简单来说，`direnv` 是为 **`.envrc`** 而生的，而对 **`.env`** 的支持则是通过其内部函数库（stdlib）实现的“兼容性扩展”。

以下是它们的核心区别与使用方式：

---

### 1. 对 `.envrc` 的原生支持 (主打功能)

`.envrc` 是 `direnv` 的核心配置文件。它的本质是一个 **Shell 脚本**。

- **加载机制**：当你进入包含 `.envrc` 的目录时，`direnv` 会启动一个子 Shell 执行该脚本，捕获其环境变量的变化，并将这些变化应用到你当前的 Shell 中。
- **功能上限**：由于是脚本，你可以写任何逻辑：

```bash
# .envrc 示例
export NODE_VERSION=18.0.0
layout node           # 自动配置 node 环境
source_env ../.common_env  # 嵌套加载其他配置
PATH_add bin          # 安全地把当前目录的 bin 加入 PATH

```

- **安全性**：必须手动运行 `direnv allow` 才能生效，防止下载的仓库自动执行恶意脚本。

---

### 2. 对 `.env` 的支持 (兼容性功能)

很多项目（如 Docker, React, Python）默认使用 `.env` 文件（简单的 `KEY=VALUE` 格式）。`direnv` 本身**不直接**读取 `.env`，但它提供了一个内置函数 `dotenv` 来加载它。

#### 如何使用？

你需要在该目录下创建一个 `.envrc` 文件，并在其中加入一行：

```bash
dotenv

```

这样 `direnv` 就会去读取同目录下的 `.env` 文件。

- **语法差异**：`.env` 通常只支持简单的键值对，不支持逻辑判断或复杂的导出命令。
- **跨平台**：`.env` 更通用，而 `.envrc` 是 Zsh/Bash 特有的。

---

### 3. 核心对比表

| 特性         | `.envrc`                       | `.env`                      |
| ------------ | ------------------------------ | --------------------------- |
| **文件类型** | 可执行 Shell 脚本              | 静态键值对文本              |
| **加载速度** | 极快 (由于有缓存机制)          | 略慢 (需解析文本)           |
| **功能深度** | 支持函数、条件判断、PATH 修改  | 仅支持环境变量定义          |
| **安全性**   | 强制需要 `direnv allow`        | 通过 `.envrc` 间接授权      |
| **适用场景** | 复杂的开发环境、编译器路径配置 | 简单的 API 密钥、数据库配置 |

---

### 4. 最佳实践：混合使用

在现代开发中，推荐的做法是将两者结合：

1. **`.env`**：存储通用的配置（如数据库 URL），方便与团队（可能使用不同 Shell 或 Docker）共享。
2. **`.envrc`**：存储个人特有的开发配置，并加载 `.env`。

**示例 `.envrc` 配置：**

```bash
# 加载通用的 .env 文件
dotenv

# 针对当前目录特有的优化
layout python  # 如果是 Python 项目，自动激活虚拟环境
export EDITOR=vim

```

---

### 💡 避坑指南：不要在 `.envrc` 中直接 `source .env`

如果你在 `.envrc` 中写 `source .env`，如果 `.env` 文件里只有 `KEY=VALUE`（没有 `export` 关键字），Shell 是不会把它们变成环境变量的。

**必须使用 `dotenv` 函数**，因为它会自动帮你处理这些键值对并导出。
